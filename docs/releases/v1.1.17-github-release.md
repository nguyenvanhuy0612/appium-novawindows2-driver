# v1.1.17

Stability release focused on long-running and stress-test scenarios. Headline: fixes a `System.OutOfMemoryException` cascade that could kill the Appium Node.js process, plus three more "PS-side state can grow unboundedly" classes of bug, plus a new session-stress regression test.

> npm: https://www.npmjs.com/package/appium-novawindows2-driver/v/1.1.17

## User-visible

### `fix(powershell)`: stream `Find-AllDescendants` on the desktop root (A14)

**Symptom observed in production** (SecureAge file-copy test against VM 192.168.196.128):

```
[PowerShell Error] Exception of type 'System.OutOfMemoryException' was thrown.
[PowerShell Raw Command]
(Find-AllDescendants -element ($rootElement) -condition (AndCondition(
    PropertyCondition(controltypeProperty, Window),
    PropertyCondition(classnameProperty, "CabinetWClass")))) -includeSelf)
```

The PS subprocess threw `OutOfMemoryException` after ~1 s, and the resulting memory pressure took down the parent Appium server process too (the parent `-NoExit` shell's prompt `PS C:\Users\admin\Desktop>` showed up in the tail of the Appium log, indicating the `appium` CLI itself had exited).

**Root cause:** the XPath descendant axis (`//Window[...]`) compiles to `Find-AllDescendants($rootElement, ...)`. The old implementation was a one-line wrapper:

```ts
$element.FindAll([TreeScope]::Subtree, $condition)
```

UIA's built-in `FindAll(Subtree)` is fast for bounded subtrees, but on the **desktop root** it materialises so much intermediate state (cached property values, COM marshalled interfaces, internal walker queues) that on a busy host it exhausts the .NET process heap and throws `OutOfMemoryException`. Under memory pressure, Windows can then kill the parent Node.js process.

**Fix:** detect when `$element` is `[AutomationElement]::RootElement` and route the search through a memory-bounded **iterative DFS via `TreeWalker`**. At any moment, only a stack-depth's worth of `AutomationElement` references plus the matches list are in memory â€” not the entire visited set. The walker respects the same `$cacheRequest.TreeFilter` as `FindAll`, so excluded views (e.g. Chrome content) stay excluded.

Non-root (window-scoped) searches keep the old fast path. As a defence in depth, a `catch [System.OutOfMemoryException]` on the fast path falls back to the streaming walker â€” best-effort, since CLR OOM is not always catchable.

**Trade-off:** the streaming walker is slower than `FindAll(Subtree)` for dense subtrees (each `GetFirstChild` / `GetNextSibling` is a separate UIA cross-process call). But it's only used when the alternative is "crash the entire Appium server", so the trade is worth it.

**Practical recommendation:** tests that don't need desktop scope should pass `appium:appTopLevelWindow` so the session attaches directly to a window. The streaming-walker path then doesn't engage at all and you keep the full speed.

### `fix(powershell)`: `Get-PageSource` is now iterative (A18)

`Get-PageSource` previously called itself recursively for every child element. PowerShell's default function-call stack ceiling is ~5000 frames; busy desktop / explorer trees blew past that and threw `ScriptCallDepthException`, crashing the PS subprocess. Driver then auto-restarts and the in-flight command is dropped â€” same blast radius as A14, different code path.

Rewritten as iterative BFS using a `(element, parentXmlElement)` queue. Stack depth is now O(1) regardless of tree depth; memory at any moment is bounded by the queue size (siblings still to visit), not by tree depth. Public signature unchanged.

### `fix(powershell)`: `$elementTable` is now FIFO-bounded at 10 000 entries (A22)

Every `findElements` / `findAll` saved every returned element into a server-side `$elementTable` dictionary that was never evicted. Long-running poll-loop sessions accumulated thousands of dead UIA COM proxies until PS hit OOM (separate from A14 â€” different root cause, same observable symptom).

Now: each insert into `$elementTable` is paired with an order-queue entry; when the table exceeds 10 000 entries, the oldest entries are evicted. Eviction is **transparent** to callers because the existing `ensureElementResolved` helper detects a missing cache entry and re-finds the element by its runtime id (one extra UIA call, no test-visible failure unless the underlying element has been destroyed).

### `fix(powershell)`: command-queue depth cap of 200

`commandQueue` is a Promise chain appended to on every `sendPowerShellCommand`. Pre-1.1.17 it had no cap â€” a client (or buggy code) sending PS calls faster than they could drain pushed the chain into unbounded growth, holding command bytes + completion-marker UUIDs + onClose handler refs, and prolonging graceful shutdown.

Now: each `sendPowerShellCommand` checks the in-flight count first; if it would exceed 200, the new command rejects immediately with `UnknownError: PowerShell command queue is full ...`. Queued commands continue to drain normally â€” the cap only refuses *new* arrivals.

### `fix(driver)`: explicit per-session cleanup in `deleteSession`

`deleteSession` now resets `commandQueue` to a fresh `Promise.resolve()`, zeroes `powerShellQueueDepth`, clears the per-command context reference, and resets `keyboardState` modifier flags. None of these affect correctness on the happy path, but they let the GC reclaim per-session closures immediately rather than waiting for a future Promise-chain root to be replaced.
