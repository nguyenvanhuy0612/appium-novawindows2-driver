# v1.1.14

Patch release: a production-triggered fix for a Robot Framework + SecureAge cascade failure (`ConnectTimeoutError` Ă— 3 during teardown of a slow-failing test case), plus several adjacent latent bugs found while auditing the PowerShell layer.

> npm: https://www.npmjs.com/package/appium-novawindows2-driver/v/1.1.14

## User-visible

### Default `powerShellCommandTimeout`: 60 s â†’ **300 s**

The driver's per-command PS timeout was bumped from 60 000 ms to 300 000 ms.

Rationale: SecureAge-style UIA queries can be ~500 ms per property; a `FindFirst` walking ~100 elements hits ~50 s, ~200 elements hits ~100 s. The previous 60 s cap was borderline for legitimate work. The longer cap is now **safe** because the kill that runs when the cap fires is non-blocking (see below) â€” so a stuck command can no longer freeze Appium's event loop while it's being killed.

If you were relying on the old 60 s ceiling, set it explicitly:

```js
'appium:powerShellCommandTimeout': 60_000,
```

### `fix(powershell)`: non-blocking `killProcessTree`

**Symptom:** in production, when a `findElement` ran past the timeout, the kill that followed froze Appium's event loop for several seconds. Any HTTP request that arrived during the freeze (notably the test teardown's `deleteSession`) saw `ConnectTimeoutError` at the TCP level. The test framework retried 3Ă— before giving up; the next test connected fine once the freeze ended.

**Root cause:** `killProcessTree` called `execSync('taskkill /F /T /PID â€¦')`. On Windows, force-killing a PS process that holds COM/UIA marshalled interfaces takes several seconds for the kernel cleanup. `execSync` blocks the Node.js event loop the whole time.

**Fix:** spawn `taskkill` detached + `unref()`. Kill runs in the background; the event loop stays unblocked. New e2e regression test [`tests/e2e/stable/killtree-nonblocking.e2e.spec.ts`](https://github.com/nguyenvanhuy0612/appium-novawindows2-driver/blob/v1.1.14/tests/e2e/stable/killtree-nonblocking.e2e.spec.ts) triggers the failure mode (hang â†’ 5 s timeout â†’ kill) and asserts `/status` stays reachable through the kill window. 3 passing.

### `fix(powershell)`: `setupRootElement` ignored `appium:appTopLevelWindow` when `appium:app` was unset

The guard was `if ((!app && !appTopLevelWindow) || (!app || app.toLowerCase() === 'none'))`. When only `appTopLevelWindow` was set, the second clause short-circuited to `true` and the function returned early with `$rootElement = $null`. The later `if (appTopLevelWindow)` branch was unreachable.

**Impact:** any session opened with just `appium:appTopLevelWindow` (a common pattern for scoping to a specific top-level window â€” used in the SecureAge Robot Framework `Open Appium Session` keyword) silently got a null root, and subsequent finds had no scope.

**Fix:** split the guard into explicit null/none/root/app/handle branches.

### `fix(powershell)`: in-flight commands during teardown reject cleanly

Previously, when `deleteSession` triggered `terminatePowerShellSession`, any command still running resolved with `''` on close. Downstream `findElOrEls` saw the empty result and threw `NoSuchElementError`; other callers got an ambiguous "empty success". Now: rejects with `NoSuchDriverError('PowerShell session terminated while command was in flight')`.

### `fix(extension)`: `executeKeys` no longer rejects `pause: 0`, `text: ''`, `virtualKeyCode: 0`

The validation counted truthy fields; falsy-but-set values were treated as absent. A legitimate `{virtualKeyCode: 0}` (or any future zero-valued virtual key) was rejected as malformed. Now: counts fields by `!== undefined`. The matching downstream `if (action.pause)` / `if (action.virtualKeyCode)` branches were updated so `pause: 0` still routes through the sleep path instead of falling through.

### `fix(winapi)`: `appium:smoothPointerMove='cubic-bezier(...)'` now actually applies the requested curve

The cubic-bezier argument extraction read `.groups?.[0]` on an unnamed-group regex. `.groups` was always `undefined`, the `?? [0,0,1,1]` fallback always won, and every cubic-bezier caller silently got the identity bezier (= linear). Switched to `match[1]`.

> Note: the broader smooth-pointer-move issue â€” no guaranteed final-frame send on short durations â€” is tracked separately under [`docs/code-review/2026-05-08.md` #15](https://github.com/nguyenvanhuy0612/appium-novawindows2-driver/blob/v1.1.14/docs/code-review/2026-05-08.md#15--bezier-easing-has-no-guaranteed-final-frame). Only the capture-group bug is fixed in 1.1.14.

## Additive (no user-visible runtime change)

### Performance: O(nÂ˛) â†’ O(n) marker scan in the PS command framing

`waitForCommandCompletion`'s `check()` ran `ctx.stdout.includes(marker)` and `ctx.stderr.includes(marker)` on every incoming chunk â€” `O(total bytes)` per chunk, `O(total bytes Ă— chunks)` overall. Tolerable for typical UIA results, degrades on large outputs (`Get-PageSource` on a complex window).

Now tracks `stdoutScanIdx`/`stderrScanIdx` and only re-scans the tail of the buffer on each chunk (backing up by `marker.length - 1` to catch markers straddling a chunk boundary). Memoises a per-side "marker seen" flag so once we've found it we don't re-scan that side at all.
