# v1.1.11 — DEPRECATED

> ⚠️ **This version is deprecated.** It hangs at session start for ~60 s before timing out. Use **[1.1.12](https://github.com/nguyenvanhuy0612/appium-novawindows2-driver/releases/tag/v1.1.12)** or later.
>
> npm tombstone: `npm deprecate appium-novawindows2-driver@1.1.11 "..."` (applied 2026-05-11)

This release was intended as a substantial PowerShell session-protocol overhaul plus a new `treatStderrAsError` capability. The protocol work is sound and lives on in 1.1.12+. The release itself shipped with a parser-level bug that wasn't caught by unit tests (which use a fake PS process). Documented here for completeness.

> npm: https://www.npmjs.com/package/appium-novawindows2-driver/v/1.1.11

## What this release contained (now in 1.1.12+)

**Session lifecycle hardening:**
- Dropped `-NoExit` from `powershell.exe` spawn args — graceful shutdown via `stdin.end()` now works instead of falling back to `SIGKILL`
- `killProcessTree` (taskkill /F /T) on the forced-shutdown path — PS children no longer orphan
- New `powerShellTerminating` flag: in-flight commands during `deleteSession` resolve silently instead of surfacing "Encountered internal error"; queued commands fast-fail with `NoSuchDriverError` instead of respawning PS
- Init / `setupRootElement` failure now kills the half-initialised session
- Child-process `error` event handler — spawn ENOENT no longer crashes Node

**Command framing protocol:**
- Per-command UUID marker (`___NOVA_END_<uuid>___`) on both stdout AND stderr
- User commands wrapped in `& { ... }` to prevent multi-line constructs from absorbing the marker lines
- Marker detection runs against the accumulated buffer, not just the chunk

**Per-command buffer isolation:**
- `powerShellStdOut` / `powerShellStdErr` removed from the driver — each command owns its own `CommandContext`
- Inter-command bytes logged at debug instead of bleeding into the next call

**Return-code handling:**
- `$LASTEXITCODE` check after the user command surfaces native-exe non-zero exits via `[NativeExit] N` on stderr
- New `treatStderrAsError` capability (default `true`); when `false`, plain stderr is treated as informational

**17 new unit tests** under `tests/unit/powershell-runtime.spec.ts` driving a fake `ChildProcess` to cover the framing, lifecycle, buffer isolation, and return-code handling.

## Why it shipped broken

The `& { ... }` wrap looked safe in isolation but breaks two things when applied to the `using namespace System.Windows.Automation` init directive:

1. **`using namespace` inside `& { }` silently hangs the PowerShell stdin parser.** The init script never completes; subsequent commands time out.
2. Even if it parsed, variables defined inside `& { }` (e.g. `$cacheRequest`, `$elementTable`, `$rootElement`) are local to that subshell and vanish — the whole driver relies on those persisting across calls.

The unit tests used a fake `ChildProcess` so the parser bug couldn't surface. E2E tests against a real PowerShell would have caught it in seconds.
