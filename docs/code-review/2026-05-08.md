## Detail

### #1 — `Math.min(a-b)` returns the difference unchanged

**File:** `lib/powershell/converter.ts:189-190`
**Severity:** CRITICAL
**Status:** 🔴 Open

```ts
const width  = Math.min(p2[0] - p1[0]);  // single-arg Math.min just returns its input
const height = Math.min(p2[1] - p1[1]);
```

`Math.min(x)` with one argument returns `x` — there is no second value to compare. The author meant `Math.abs(...)` so a `Rect(point1, point2)` selector built from two points produces a positive width/height regardless of point order. With negative dims `[System.Windows.Rect]::new(...)` either throws on the PS side or builds an empty rect.

**Repro**: pass a Windows-Automation selector containing `Rect(@(10,10), @(5,5))` — the resulting condition silently doesn't match anything sensible.

**Fix:** `Math.abs(p2[0] - p1[0])`. While here, also coerce the regex group strings to numbers with `Number(...)` since `p1`/`p2` are arrays of strings.

### #2 — Dead-code optional chain on `firstMatch`

**File:** `lib/driver.ts:205`
**Severity:** HIGH
**Status:** 🔴 Open

```ts
if (typeof w3cCaps?.firstMatch?.some['appium:appTopLevelWindow'] === 'number') {
    w3cCaps.firstMatch['appium:appTopLevelWindow'] = w3cCaps.firstMatch['appium:appTopLevelWindow'].map(String);
}
```

`firstMatch` in the W3C protocol is an **array** of capability dicts. `.some` on an array is `Array.prototype.some` (a function), so `someFunction['appium:…']` is `undefined`, `typeof undefined === 'number'` is always false, and the branch never runs. As a result, a numeric `appTopLevelWindow` inside `firstMatch[i]` is never coerced to string the way the alwaysMatch sibling does.

**Fix:** iterate `firstMatch` and coerce per entry, or normalize the whole array before calling `super.createSession`.

### #3 — `trySetForegroundWindow(NaN)` after name-based lookup

**File:** `lib/commands/app.ts:85, 105`
**Severity:** HIGH
**Status:** 🔴 Open

```ts
const handle = Number(nameOrHandle);            // line 85; NaN if the arg is a window name
…
if (elementId.trim() !== '') {                  // name branch
    …
    trySetForegroundWindow(handle);             // line 105 — called with NaN
    return;
}
```

When `nameOrHandle` is a name like `"Notepad"`, `Number(...)` is `NaN`. The foreground call is a no-op, so after switching by name the target window often stays in the background — exactly what the call is meant to prevent.

**Fix:** after resolving the element by name, read its native window handle (UIA `Property.NATIVE_WINDOW_HANDLE`) and pass that to `trySetForegroundWindow`.

### #4 — PS injection in file commands

**File:** `lib/commands/file.ts:11, 28, 42`
**Severity:** HIGH
**Status:** 🔴 Open

```ts
const command = `
$targetPath  = '${escapedPath}';
$base64String = '${base64Data}';   // ← unescaped client input
…`;
```

Only `escapedPath` doubles single quotes. `base64Data` is a public W3C input that gets interpolated raw — a client sending `';iex(<payload>);'` as the "base64" string can run arbitrary PowerShell in the persistent session.

**Fix options** (any one is sufficient, prefer the first):
1. Validate `base64Data` matches `^[A-Za-z0-9+/=\s]*$` before interpolation (rejects anything that can break out of `'…'`).
2. Escape via the same `replace(/'/g, "''")` rule already used for paths.
3. Pipe via stdin or a base64-safe variable rather than literal interpolation.

`pullFolder` only interpolates `$targetPath` (already escaped) — lower risk, but worth auditing in case a future caller adds untrusted input.

### #5 — Pattern handlers bypass `resolvePatternElement`

**File:** `lib/commands/extension.ts:421, 426, 437, 442, 446, 460, 482`
**Severity:** HIGH
**Status:** 🔴 Open

`patternIsMultiple`, `patternGetSelectedItem`, `patternGetAllSelectedItems`, `patternAddToSelection`, `patternRemoveFromSelection`, `patternSetValue`, `patternGetValue` all read `element[W3C_ELEMENT_KEY]` directly. They skip the `resolvePatternElement` + `ensureElementResolved` chain that `patternSelect`/`patternToggle` use.

Result: if the element has been evicted from the PowerShell session table (after a session reattach, or because the runtime ID went stale), the call surfaces a raw PowerShell `null-valued expression` error instead of a clean W3C `NoSuchElementError`. Inconsistent + bad UX for clients.

**Fix:** route every pattern handler through the same resolver helper.

### #6 — Raw runtime ID interpolation in `ELEMENT_TABLE_GET`

**File:** `lib/powershell/elements.ts:270`
**Severity:** HIGH (defense-in-depth) / MEDIUM (current real-world risk)
**Status:** 🔴 Open

```ts
const ELEMENT_TABLE_GET = pwsh$ /* ps1 */ `$elementTable['${0}']`;
```

The `pwsh$` template just interpolates the value into the `'…'` literal. Runtime IDs are returned to W3C clients and accepted back. A client sending `1.2'; <payload>; '` as an element ID would inject. We're not aware of an exploit today (the W3C key is not surfaced to untrusted parties in our pipeline), but the trust boundary is at the W3C input.

**Fix:** validate IDs match `^\d+(\.\d+)*$` in `ensureElementResolved`, or use a properly-quoted PS variable substitution in the template.

### #15 — Bezier easing has no guaranteed final frame

### #21 — No auto-recovery after PowerShell session crashes

**File:** `lib/commands/powershell.ts` (`sendPowerShellCommand` + `ensurePowerShellSession`)
**Severity:** HIGH (E2E test stability)
**Status:** 🟢 Fixed (this branch)

**Trigger observed in production**: a Robot Framework test on VM 192.168.196.129 ran an `execute('powerShell', ...)` script that called `(New-Object -ComObject Shell.Application).Windows() | …Quit()` followed by `Invoke-Item "C:\<folder>"`. The next UIA `findElements` query traversed the desktop tree, hit a half-shutdown explorer COM proxy, and the whole `powershell.exe` host process exited with code 5 (`ERROR_ACCESS_DENIED`).

The driver's existing `powerShell.on('exit', …)` handler (added in a previous patch) detected the death cleanly — Node didn't crash. But every subsequent command, including the test teardown, then failed with:

```
UnknownError: PowerShell session is not available or closed
    at ensureSessionReady (lib/commands/powershell.js:63:15)
```

The whole Robot suite cascade-failed because there's no recovery path short of `deleteSession` + re-`createSession`.

**Fix:** added `ensurePowerShellSession(driver)` helper that runs at the start of every `sendPowerShellCommand`:
- If `this.powerShell` is alive (`exitCode === null && stdin.writable`), no-op.
- Otherwise, spawn a new session via `startPowerShellSession.call(driver)`.
- Concurrent callers (a burst of queued commands all observing a dead session) await the same `powerShellRestartPromise` field so we restart exactly once.
- Restart is done **before** the command queue, since `startPowerShellSession` internally calls `sendPowerShellCommand` for the init script — putting the restart inside the queue would deadlock on the outer queue entry.

**Trade-offs:**
- The element table (`$elementTable`) is empty after restart. Any cached element IDs the test code held onto surface as `NoSuchElement` on the next access — the correct behaviour, since the underlying UIA elements may have been destroyed by whatever killed PS in the first place.
- Capability-driven session state is rebuilt: `appWorkingDir`, the helper scripts, and `setupRootElement` (which re-attaches to the app/window from `caps.app` / `caps.appTopLevelWindow`) all run again. `prerun` is **not** re-run (avoids repeating side effects).
- If the underlying app crashed too and `setupRootElement` fails during restart, the error propagates back to the caller as a clean `UnknownError`. The next call will try restart again (the promise is reset in `finally`).

**Verification:**
- `npm run build`: clean.
- `npm test`: 880 passing, 0 failing.
- On-VM verification deferred until next deploy.

### #22 — Undeclared runtime dependencies (asyncbox + teen_process)

**File:** `package.json` + `lib/commands/screen-recorder.ts`
**Severity:** CRITICAL — published 1.1.9 fails to load on fresh hosts
**Status:** 🟢 Fixed in 1.1.10

**Reproduction (1.1.9 from npm)**:
1. `appium driver install --source=npm appium-novawindows2-driver` on a clean Win VM
2. Start Appium

```
[Appium] Could not load driver 'novawindows2', so it will not be available.
[Appium] Error: Cannot find module 'asyncbox'
Require stack:
- C:\Users\admin\.appium\node_modules\appium-novawindows2-driver\build\lib\commands\screen-recorder.js
- C:\Users\admin\.appium\node_modules\appium-novawindows2-driver\build\lib\commands\index.js
- C:\Users\admin\.appium\node_modules\appium-novawindows2-driver\build\lib\driver.js
```

**Root cause**: `lib/commands/screen-recorder.ts` does eager top-level `import { waitForCondition } from 'asyncbox'` and `import { SubProcess } from 'teen_process'`. Both packages were not declared in `package.json` "dependencies" — they happened to resolve via the dev tree on the developer machine (transitively through `appium`'s own deps), but a fresh `appium driver install` only installs declared deps. Both are also pulled into the eager-load path via `lib/commands/index.ts` re-exporting `screen-recorder`, so the failure takes down the whole driver, not just the recording feature.

**Compounding constraint (Win-ARM64)**: this driver targets Windows-ARM64 too. `ffmpeg-static` is already in `optionalDependencies` because no prebuilt FFmpeg binary exists for ARM64. The recording stack should be **fully optional** so an ARM install (or any host that doesn't need recording) skips it cleanly. Adding `asyncbox`/`teen_process` to required `dependencies` would technically work (both are pure JS, `os: any, cpu: any`, no native code) but defeats the "recording is optional" design — same eager-load failure mode, just shifted.

**Fix applied**:

1. **`package.json`**: declared `asyncbox` and `teen_process` as `optionalDependencies` alongside `ffmpeg-static` (matching the existing optional-recording pattern).
2. **`lib/commands/screen-recorder.ts`**: removed top-level imports of `asyncbox` and `teen_process`. Added a `loadRecordingDeps()` helper that lazy-`require()`s them inside the recording functions, throwing a clean `Error('Screen recording is not available: optional dependency X is not installed. Install with: npm i asyncbox teen_process ffmpeg-static')` if missing. The `_process` field type was loosened to `any` since the static `SubProcess` type is no longer imported. `appium/support` imports stay eager — `appium` is a peer dep that's always satisfied when the driver is loaded by Appium.

**Behavior matrix after fix**:

| Scenario | Result |
| :--- | :--- |
| Win-x64 with full recording stack installed | ✅ recording works |
| Win-x64, npm hiccup → `asyncbox` missing | ✅ driver loads; `windows: startRecordingScreen` throws clean error |
| Win-ARM64, recording deps intentionally skipped | ✅ driver loads; recording calls throw clean error |
| Win-ARM64, all deps installed but no FFmpeg binary for ARM | ✅ driver loads; recording call throws "ffmpeg-static binary missing or not supported on this architecture" |

**Verification**:
- `npm run build` clean.
- `npm test` 880/880 passing.
- Pending: re-publish to npm as 1.1.10, redeploy on VM 132 to confirm `appium driver install --source=npm` succeeds.
