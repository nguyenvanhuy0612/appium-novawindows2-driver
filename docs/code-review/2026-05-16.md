## Detail — A1 to A2 (the production trigger)

### A1 — `killProcessTree` blocks the event loop

**File:** `lib/commands/powershell.ts:89-104`
**Severity:** HIGH (production failure trigger)
**Status:** 🟢 Fixed (686027a)

**Production trigger.** Robot Framework + SecureAge, QAOFF-309 TC_009: a `findElement(name='Shell Folder View')` against a SecureAge dialog ran past the 60 s `powerShellCommandTimeout`. The timeout handler called `killProcessTree(pid)` → `execSync('taskkill /F /T /PID …')`, which is **synchronous and blocks Node.js's event loop** while taskkill runs.

On Windows, killing a PS process that's mid-UIA-COM call can take several seconds because the COM runtime needs to tear down marshalled interfaces before the kernel declares the process exited. During those seconds, Appium can't accept TCP connections. The Robot teardown that immediately follows sees `ConnectTimeoutError` × 3 (one per teardown step, 20 s each), and TC_010 runs cleanly once the kill finally completes.

**Fix:** spawn `taskkill` detached + `unref()` (fire-and-forget). Event loop stays unblocked; `driver.powerShell = undefined` is set immediately, and the auto-restart path kicks in on the next command.

```ts
// Before:
execSync(`taskkill /F /T /PID ${pid}`);

// After:
const child = spawn('taskkill', ['/F', '/T', '/PID', String(pid)], {
    stdio: 'ignore',
    detached: true,
});
child.unref();
```

**Verification:** new regression test `tests/e2e/stable/killtree-nonblocking.e2e.spec.ts` triggers a hang with a 5 s `powerShellCommandTimeout` and polls `/status` at 100 ms intervals through the kill window. Pre-fix: probes TCP-timeout. Post-fix: every probe under STATUS_CONNECT_TIMEOUT_MS (1.5 s). 3 passing.

### A2 — `setupRootElement` ignores `appTopLevelWindow`

**File:** `lib/commands/powershell.ts:560-592`
**Severity:** HIGH (silent failure)
**Status:** 🟢 Fixed (686027a)

```ts
// Before:
if ((!app && !appTopLevelWindow) || (!app || app.toLowerCase() === 'none')) {
    await sendPowerShellCommand.call(this, NULL_ROOT_ELEMENT);
    return;
}
```

Trace for `app=undefined, appTopLevelWindow="12345"`:
- 1st clause: `(true && false) = false`
- 2nd clause: `(!app || …) = true` (short-circuit, doesn't evaluate `app.toLowerCase()`)
- Whole expression: `true` → early-return with `$rootElement = $null`

The `if (appTopLevelWindow)` branch later in the function is unreachable. Sessions opened with just `appium:appTopLevelWindow` silently get a null root, and later finds fall back to whatever happens with `$rootElement=$null` in PS.

**Concrete repro path:** SecureAge tests (`keywords/base_swin.resource` → `Open Appium Session`) remove `appium:app` from caps and add `appium:appTopLevelWindow` for sub-sessions scoping to a specific window. These all hit the buggy branch.

**Fix:** split into explicit branches for each case (null/none/root/app/handle).

## Detail — A3 to A6 (regex / type-guard bugs)

### A3 — `cubic-bezier` capture-group read

**File:** `lib/winapi/user32.ts:700-701`

```ts
// Before:
const bezierArgs = /\((.*?)\)/.exec(easingFunction)?.groups?.[0]
    .split(',').map((n) => parseFloat(n.trim())) ?? [0, 0, 1, 1];
```

The regex has an unnamed capture group, so `.groups` is `undefined`. `?.groups?.[0]` short-circuits to undefined, and `?? [0, 0, 1, 1]` (identity bezier = linear) always wins. Every `appium:smoothPointerMove='cubic-bezier(...)'` caller silently got linear interpolation.

**Fix:** read the first numbered capture (`match[1]`).

### A4 — `PSCultureInfo` overload + emit

**File:** `lib/powershell/common.ts:131-135`

Two bugs in one constructor:

1. The type guard `typeof nameOrCulture !== 'string' || (typeof nameOrCulture === 'number' && nameOrCulture < 0)` fires on the first clause for every non-string, including the documented number overload. `new PSCultureInfo(1033)` always throws — the number overload is unreachable.

2. The boolean append `${useUserOverride !== undefined ? `$${useUserOverride}` : ''}` produced PS like `::new('en-US'$true)` — no comma between args, PS parse error.

**Fix:** check `isString || isNonNegativeNumber` explicitly, and prepend `, ` to the boolean argument.

### A5 — `converter.ts` Rect / Point flat-capture handling

**File:** `lib/powershell/converter.ts:165-220`

Each `RECT_PARAMETER_REGEX_C{1,3,4,5}` is built by embedding inner Point / Size / Vector / floating-point matchers verbatim, so their unnamed capture groups become flat numbered captures on the outer regex. Two pre-fix bugs lived here:

1. The `replaceAll` callbacks declared too few args (`(_, point, size) => …` when the regex had 4 captures). `point` was actually the X coordinate as a string; `size` was the Y coordinate; the W and H captures were dropped.
2. The body then re-ran the inner regex with `.groups` (always undefined for unnamed groups) → fallback `[]` → `Number(undefined) = NaN`.

Net: every `Rect(Point, Size)` / `Rect(Point, Point)` / `Rect(Point, Vector)` / `Rect(Size)` in a `-windows uiautomation` selector parsed to `Rect(NaN, NaN, NaN, NaN)`.

**Fix:** rewrote each callback to enumerate the actual flat captures and call `Number(...)` directly. C2 (4 plain floating-point args) was already correct; left untouched.

### A6 — `PropertyCondition` swapped enum bounds

**File:** `lib/powershell/conditions.ts:89,101`

```ts
// In the OrientationTypeProperty branch:
} else if (!(value instanceof PSInt32) || Number(value.toString()) < 0
    || Number(value.toString()) >= Object.keys(AutomationHeadingLevel).length) {
    throw e;
}

// In the AutomationHeadingLevelProperty branch:
} else if (!(value instanceof PSInt32) || Number(value.toString()) < 0
    || Number(value.toString()) >= Object.keys(OrientationType).length) {
    throw e;
}
```

The bound check in each branch reads the *other* enum's length. With different cardinalities, legitimate integer values for one are rejected based on the other's size.

**Fix:** swap to use the correct enum in each branch.

## Detail — A7 to A11 (powershell.ts robustness)

### A7 — In-flight teardown rejection

**File:** `lib/commands/powershell.ts:215-220`

Pre-fix: when `deleteSession` triggers `terminatePowerShellSession`, in-flight `waitForCommandCompletion` sees `powerShellTerminating=true` on `'close'` and resolves with `''`. Downstream `findElOrEls` checks `if (!elId) throw NoSuchElementError`, surfacing the teardown as a NoSuchElementError to the test framework. Other callers got an ambiguous "empty success".

**Fix:** reject with `NoSuchDriverError('PowerShell session terminated while command was in flight')` instead. Two unit tests in `tests/unit/powershell-runtime.spec.ts` updated to assert the new behaviour.

### A8 — `powerShellCommandTimeout` default 60 s → 300 s

**Files:** `lib/driver.ts:230`, `lib/commands/powershell.ts:384`

Rationale: SecureAge UIA queries can be ~500 ms per property (see [`secureage_slow_uia` memory note]). A FindFirst that walks ~100 elements hits ~50 s; ~200 elements hits ~100 s; pathological scopes can exceed that. 60 s was borderline; 120 s still occasionally clipped real work. 300 s is a generous ceiling that should only fire on a genuine hang.

Longer timeouts are now cheap: `killProcessTree` is non-blocking (A1), so a stuck command no longer freezes Appium's event loop while it's being killed. The only cost of a higher cap is a slower failure when a command genuinely hangs.

Tests / specific callers that care about a tighter ceiling should pass `appium:powerShellCommandTimeout` in caps.

### A9 — O(n²) marker scan

**File:** `lib/commands/powershell.ts:140` (was)

Pre-fix `check()` ran `ctx.stdout.includes(marker)` and `ctx.stderr.includes(marker)` on every incoming chunk — that's `O(total bytes)` per chunk, `O(total bytes × chunks)` overall. Tolerable for typical UIA results (sub-MB) but degrades with large outputs (`Get-PageSource` on a complex window).

**Fix:** track `stdoutScanIdx`/`stderrScanIdx` on the `CommandContext`. Each new chunk only re-scans from `max(0, prevIdx - (marker.length - 1))` to handle markers straddling a chunk boundary. Worst-case scan is now `O(total bytes)`, not `O(total bytes × chunks)`. Memoise `stdoutHasMarker`/`stderrHasMarker` flags so once we've seen the marker we don't re-scan that side at all.

### A10 — `||` → `??` + validation for the timeout cap

**File:** `lib/commands/powershell.ts:384`

`||` treated `0` and `NaN` as "use the default". Switched to `??` plus an explicit `typeof === 'number' && Number.isFinite() && > 0` validation; if the cap is anything else, fall back to the new 300 s default. Mirrored in `driver.ts`'s `createSession`.

### A11 — `Set-Location` quote escape

**File:** `lib/commands/powershell.ts:524`

```ts
// Before:
await sendPowerShellCommand.call(this, `Set-Location -Path '${expandedDir}'`);

// After:
const psEscaped = expandedDir.replace(/'/g, "''");
await sendPowerShellCommand.call(this, `Set-Location -Path '${psEscaped}'`);
```

PS single-quoted-string escape is doubled single quote. Without the escape, an `appWorkingDir` containing `'` (rare but legal on Windows) would break the command.

### A12 — `pullFolder` temp-zip cleanup

**File:** `lib/commands/file.ts:41-58`

Wrapped the `Compress-Archive` + `ReadAllBytes` + base64-encode block in PS `try { … } finally { Remove-Item -EA SilentlyContinue }`. Pre-fix `Remove-Item` was unreachable when an exception interrupted the script, leaving zips in `$env:TEMP`.

### A13 — `DeferredStringTemplate.format` cleanup

**File:** `lib/util.ts:69-76`

Pre-fix loop ran `for (let i = 0, k = 0; i < literals.length; i++, k++) { out[k] = literals[i]; out[++k] = args[substitutions[i]]?.toString(); }`. On the last literal, `substitutions[i]` is `undefined`, so `args[undefined]` is `undefined`, and `out` got a trailing `undefined`. `Array.join('')` stringifies `undefined` to empty, so the result was correct — but the index math was confusing.

**Fix:** rewrote as `for (let i = 0; i < literals.length; i++) { out.push(literals[i]); if (i < substitutions.length) out.push(args[substitutions[i]]?.toString() ?? ''); }`.

## Detail — A14 (Find-AllDescendants OOM)

### A14 — `Find-AllDescendants` on the desktop root throws `OutOfMemoryException`

**File:** `lib/commands/functions.ts` (Find-AllDescendants)
**Severity:** CRITICAL — can take down the parent Appium server, not just the PS subprocess
**Status:** 🟢 Fixed in 1.1.17

**Production trigger.** SecureAge file-copy test against VM 192.168.196.128, ~12 hours after the 1.1.14 deploy. Session opened with `app: 'Root'` (desktop), test launched explorer at `C:\EncryptFolder`, then issued `findElements(//Window[@ClassName='CabinetWClass']//Button[@Name='Minimize'])`. The XPath compiler emitted:

```ps1
Find-AllDescendants -element ($rootElement) -condition (
  [AndCondition]::new(
    [PropertyCondition]::new([AutomationElement]::controltypeProperty, [ControlType]::Window),
    [PropertyCondition]::new([AutomationElement]::classnameProperty, "CabinetWClass")
  )
) -includeSelf
```

After 1004 ms the PS subprocess emitted:

```
[PowerShell Error] Exception of type 'System.OutOfMemoryException' was thrown.
```

…then the parent Appium Node.js process exited too. The `-NoExit` PS shell that launched `appium --relaxed-security ...` returned to its prompt and wrote `PS C:\Users\admin\Desktop>` as the last line of `appium_server.log`.

**Root cause.** The pre-fix implementation was a one-line passthrough to UIA:

```ts
function Find-AllDescendants { ... return $element.FindAll($scope, $condition); }
```

UIA's `FindAll([TreeScope]::Subtree, ...)` is fast for bounded subtrees, but on the **desktop root** it materialises so much intermediate state (cached property values, COM marshalled interfaces, internal walker queues) that on a busy host the .NET process heap is exhausted. The `OutOfMemoryException` is non-catchable in some CLR conditions; even when caught, the runtime is in a degraded state.

**Why this didn't surface before 1.1.14.** 1.1.13 had a 60 s `powerShellCommandTimeout` default; the slow path through `FindAll(Subtree)` likely hit the timeout first and surfaced as a normal timeout rejection (whose blocking `taskkill` then triggered the *other* production cascade fixed in A1 / 686027a). 1.1.14 raised the default to 300 s, which gave the search enough rope to actually run out of memory.

**Fix.** Detect `$element.Equals([AutomationElement]::RootElement)` and route those calls through a new `Find-AllDescendantsStreaming` function that does an iterative DFS via `TreeWalker`:

```ps1
$walker = [TreeWalker]::new($cacheRequest.TreeFilter);
$matches = New-Object System.Collections.Generic.List[AutomationElement];
# … push root's children onto a stack, pop one at a time, test the
# condition, push their siblings and first-child. Memory at any moment
# is O(tree depth + number of matches), not O(total nodes visited).
```

The walker uses the same `$cacheRequest.TreeFilter` as `FindAll`, so excluded views stay excluded. Non-root searches keep the fast path. As defence-in-depth, `catch [System.OutOfMemoryException]` on the fast path also falls back to the streaming walker (best-effort — CLR OOM is not always catchable).

**Trade-off.** The streaming walker is slower than `FindAll(Subtree)` because each `GetFirstChild` / `GetNextSibling` is a separate UIA cross-process call. Acceptable since:
- It only runs when the alternative is "crash the entire Appium server".
- Tests that don't need desktop scope should pass `appium:appTopLevelWindow` so the session attaches directly to a window — the streaming path never engages.

**Tangential learning** (saved to memory as `pwsh_btoa_latin1.md`): while writing this fix, an em-dash `—` accidentally landed in a PS comment inside the `pwsh \`...\`` template. That made `btoa()` throw at module-load time, which surfaced as `ERR_MODULE_NOT_FOUND` on the *importing* file (`lib/commands/powershell`) rather than as a useful error. The `pwsh` tagged template is strictly Latin-1; comments inside must avoid non-ASCII punctuation.
